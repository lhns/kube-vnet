WARNING: this cluster is Kubernetes {{ $kubeMajor }}.{{ $kubeMinor }}, older than 1.30, so the three
ValidatingAdmissionPolicies kube-vnet ships were NOT installed. They are the
admission-time boundary that protects:

  * operator-owned `kube-vnet.system/*` labels from user tampering
  * the reserved VirtualNetwork names `namespace` and `cluster`
  * `kube-vnet/net.*` join-label values (bad values are admitted instead)

Without them, a user with edit rights can forge those labels. The operator's
drift correction still reverts the change within a reconcile cycle, but that is
a race, not a boundary — see docs/security/threat-model.md (assumption A-2,
finding F-02).

Upgrade to Kubernetes >= 1.30 to close this gap.

{{- end }}

The VirtualNetwork CRD is registered. Define a network and label pods to join it:

  kubectl get vnet -A
  kubectl get networkpolicy -A -l kube-vnet.system/managed-by=kube-vnet

Sample manifests are in the project repo at
https://github.com/lhns/kube-vnet/tree/main/config/samples

On `helm uninstall`, a pre-delete hook removes operator-managed
NetworkPolicies (`-l kube-vnet.system/managed-by=kube-vnet`) before tearing down
the controller — without it, the deny-all baselines would keep enforcing
after the operator is gone. CRDs and CRs (with `helm.sh/resource-policy:
keep`) survive uninstall as before. To skip the hook (e.g. for a controlled
migration where you want to preserve the existing policy set):

  helm uninstall {{ .Release.Name }} -n {{ .Release.Namespace }} --no-hooks
  # then clean up manually when ready:
  kubectl delete networkpolicy -A -l kube-vnet.system/managed-by=kube-vnet

See ADR 0036 for the full lifecycle.

WARNING: operator.clusterBaseline.ingressIsolationLevel is "pod" — the
strictest preset. Every managed namespace gets the deny-all ingress
baseline, and pods receive ingress ONLY via explicit vnet membership
(label, binding, or baseline membership). Workloads that previously
relied on Kubernetes default-allow will lose ingress connectivity until
they are joined to a vnet.

Namespaces in operator.disabledNamespaces (default: kube-system) are exempt —
keep kube-system there unless you have a deliberate reason to isolate the
control plane. Enrolling it is supported: the chart then renders the CoreDNS
carve-out so cluster DNS keeps working (see ADR 0042).

{{- end }}
